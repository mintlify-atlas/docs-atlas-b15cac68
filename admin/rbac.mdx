## Resoluciأ³n de rol durante el login

Cuando un usuario inicia sesiأ³n, `fetchProfile` en `AuthContext.jsx` construye el objeto de usuario con sus permisos siguiendo este orden de prioridad:

<Steps>
  <Step title="Super Admin bypass">
    Si `authUser.app_metadata.is_super_admin === true`, se asigna directamente:
    ```jsx

    ```
    No se realizan mأ،s consultas a la base de datos.
  </Step>
  <Step title="Obtener empleado y rol relacional">
    Se consulta `employees` filtrando por email. Si el empleado tiene `role_id`, se hace un segundo fetch a la tabla `roles` para obtener el objeto completo del rol.
    ```jsx
    // Prioridad: rol relacional (FK) > campo role legado > employee_type
    const roleName = employeeData.roles?.name || employeeData.role || employeeData.employee_type
    ```
  </Step>
  <Step title="Validar acceso">
    - Si `is_active === false` â†’ logout inmediato.
    - Si el rol tiene `web_access === false` â†’ logout inmediato.
  </Step>
  <Step title="Buscar permisos de mأ³dulos">
    Se realizan hasta tres intentos con `ilike` en `role_modules`:

    1. **Por nombre de rol** (`roles.name` o campo `role` legado)
    2. **Por posiciأ³n/cargo** (si es diferente al rol y el intento 1 falla)
    3. **Por categorأ­a genأ©rica** (ver siguiente secciأ³n)
  </Step>
  <Step title="Fallback a categorأ­a genأ©rica">
    Si ningأ؛n intento anterior devuelve permisos, se extrae la categorأ­a del cargo:

    ```jsx

    if (name.startsWith('GERENTE'))     return 'GERENTE'
    if (name.startsWith('SUPERVISOR'))  return 'SUPERVISOR'
    if (name.startsWith('COORDINADOR')) return 'COORDINADOR'
    if (name.startsWith('ANALISTA'))    return 'ANALISTA'
    ```

    Luego se busca esa categorأ­a en `role_modules`. Esto permite configurar una entrada `JEFE` que aplique a todos los `JEFE DE OPERACIONES`, `JEFE DE VENTAS`, etc.
  </Step>
  <Step title="Fallback administrativo">
    Si el rol efectivo es `ADMIN`, `SUPER ADMIN` o `ADMINISTRADOR GENERAL` y no existe ninguna entrada en `role_modules`, se asigna acceso total:
    ```jsx

    ```
  </Step>
</Steps>

## Super Admin: bypass completo

El flag `is_super_admin` vive en `app_metadata` de Supabase Auth, no en la tabla `employees`. Esto lo hace **inmutable desde el navegador** â€” solo puede modificarse con la `service role key`.

```sql
-- Otorgar Super Admin
UPDATE auth.users
SET raw_app_meta_data = raw_app_meta_data || '{"is_super_admin": true}'::jsonb
WHERE LOWER(TRIM(email)) = 'usuario@empresa.com';

-- Revocar Super Admin
UPDATE auth.users
SET raw_app_meta_data = raw_app_meta_data - 'is_super_admin'
WHERE LOWER(email) = 'usuario@empresa.com';
```

<Warning>
  Ejecutar estas sentencias requiere la `service role key`. Nunca exponga esta clave en el frontend ni en variables de entorno pأ؛blicas.
</Warning>

## Actualizaciones en tiempo real

| Tabla escuchada | Condiciأ³n de recarga | Efecto |
|---|---|---|

```jsx
    const changedRole = payload.new?.role_name || payload.old?.role_name

        changedRole.toUpperCase() === user.role.toUpperCase()) {

    }
  })
  .subscribe()
```
